About the project

How it was built,
and why.

A personal experiment in retrieval and tool use over a notoriously dense rulebook. Not a polished product — answers can be wrong, citations can drift, and the system struggles with questions that chain several lookups. The goal is to find out how close a cheaper model with the right scaffolding can get to a frontier model on this task, and to publish where it falls short.

01
Motivation

A 1M-token rulebook and
a cost question.

Advanced Squad Leader has one of the most complex rulebooks of any game ever published — over a million tokens, densely cross-referenced, with dozens of tables, exceptions, and numerical calculations. Looking up rules mid-game is slow and error-prone even for experienced players.

A frontier model like gpt-5.4 answers most rules questions well, but at a price point that is hard to justify for a hobby tool. Mini-class models cost a small fraction of that. The experiment is whether the right retrieval and tool scaffolding can close enough of the gap to make a cheap model practically useful at the table.

This is a personal project I built for my own use while learning the game. The eval results, including the failures, are published on the evals page.

02
Architecture

FastAPI, the OpenAI
Responses API, and a WebSocket.

The backend is FastAPI behind nginx on Digital Ocean. Chat streams over a persistent WebSocket so the user sees tokens as they arrive and can interrupt mid-answer. Each query travels the same four-stage pipeline:

  1. i
    Retrieve & generate
    The question is sent to the OpenAI Responses API with a file_search tool over a vector store of the rulebook. Up to 20 chunks are pulled into a structured system prompt that tells the model to cite section numbers. Tokens stream back over the WebSocket along with TTFT, RAG latency, total time, token counts, and per-query cost.
  2. ii
    Ground in the source
    Citations in the response are rendered as clickable links that open an in-browser PDF viewer at the exact rulebook page, using a pre-built section-to-page index (section_pages.json). The user can check every claim against the source without leaving the chat.
  3. iii
    Evaluate
    A growing eval set of 200+ rules questions runs against the current model. A zero-shot LLM judge scores each response Pass / Fail / Needs Review against an expected answer; flagged rows are reviewed manually. Results are broken down by question type and published on the evals page.
  4. iv
    Curate & retrain
    Every chat is logged. Promising Q&A pairs are triaged through a review UI into eval sets or SFT training data. Strong examples feed the next iteration; mistakes become regression tests.
03
Attach a .vsav

Pasting in your
board state.

VASL is the dominant online client for ASL. Every game on VASL can be written to a .vsav save file — a complete, machine-readable snapshot of the board: which boards are in play, their orientation, every counter, its hex, its stacking order, the face it shows, and any SSR-level terrain transforms.

The chat has a 📎 .vsav button on both the authenticated chat and the public demo. Drop a save file in and the server parses it directly — no screenshot involved — and appends a plain-text BOARD STATE block to your question. Hex IDs, units, state flags (BROKEN, concealed, HIP), per-unit markers (DM, Prep Fire, Melee, Foxhole: in), and the terrain of every occupied hex are all included.

The model is told to treat that block as ground truth: it is exact, machine-parsed data from the save file, so it wins over any screenshot if the two disagree about a unit's identity or position. That removes a whole class of vision errors — misread counter labels, wrong stacking, mistaken terrain — from the answer.

One file picker, no manual setup, no schema to learn. Open the demo and try it with one of your own saves.

04
Deterministic

Where rules end and
arithmetic begins.

Many ASL questions split cleanly in two: a retrieval question (which rule applies here?) and an arithmetic question (what does that rule compute to?). LLMs are good at the first and unreliable at the second. The same model that correctly cites A7.21 PBF and B27 entrenchment TEM will, on the same turn, quietly drop a +1 DRM or round an IFT column the wrong way.

The fix is to take the math out of the model and put it in code. The assistant ships four deterministic function-calling tools that the LLM is required to invoke whenever a question reduces to arithmetic, and to never derive by hand:

  1. i
    ift_odds · IFT lookup
    Column + DRM in, break / pin / casualty odds out, read from the Infantry Fire Table. Also available as the /ift page for direct use.
  2. ii
    ift_attack · Full attack builder
    The whole A7.2–.36 pipeline: per-unit FP, range halving and doubling, FP column, itemized DRM, auto cowering, break / pin / casualty odds against a target morale or a vehicle's ★ line. Used whenever the user describes a situation in text.
  3. iii
    resolve_attack · Save-driven fire
    Given an attached .vsav and a firing and target hex, the server computes the entire fire attack from the parsed save: which units fire, final FP, range / PBF, every DRM line item, and the IFT result. No values from memory; every modifier is derived from the file and an itemized ledger is returned to the model verbatim.
  4. iv
    resolve_cc · Save-driven Close Combat
    Given a hex and an attacker side, the server computes both sides of a simultaneous CC: the per-unit FP ledger, DRMs, CCT column after odds, the DR thresholds for elimination and Casualty Reduction, and Melee state. A11 is exactly where hand derivation breaks; this removes the chance to.

The model still reads the rules, picks the right tool, and writes the explanation; the tool supplies the numbers and the per-line derivation, which the model is instructed to surface to the user verbatim — along with its assumptions and warnings (LOS not modeled, no Ambush dr, etc.). The result is an answer where the prose can be judged on rules interpretation, and the arithmetic is auditable line by line.

05
Hardness

Where this domain bites the
standard RAG playbook.

Scale
At over a million tokens, the rulebook is far longer than any context window can hold. Naive "stuff it in the prompt" approaches don't apply.
Cross-references
Rules routinely reference other rules by section. A complete answer often requires retrieving from several non-adjacent sections in one query.
Multimodal content
The book carries hex maps, counter diagrams, and fire tables. Pure text retrieval loses visual context some rules depend on; vision input and a fixed terrain legend partially compensate.
Precision
A "mostly right" answer that misses one exception or DRM produces incorrect gameplay. Vague answers have no value at the table.
Calculation
Many questions chain lookups across multiple tables (terrain, unit stats, IFT columns) that retrieval alone cannot resolve — the deterministic tools above exist for this reason.

About ASL

A 1985 hex-and-counter
tactical wargame.

Advanced Squad Leader is a tactical-level hex-and-counter wargame covering WW2 and the Korean War, originally published by Avalon Hill in 1985 and now by Multi-Man Publishing. It is widely regarded as the most comprehensive squad-level wargame ever published, and most play today happens online over VASSAL and VASL.
